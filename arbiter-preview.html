Previewing Arbiter 0.1 beta.

Arbiter is Ephemerent's verifier-grounded coding model route. It pairs a strong coding LLM with latent reasoning so software agents can propose, check, repair, and keep evidence before returning a final answer.

PrincipleThe verifier decides

A normal coding model can write a plausible patch. Arbiter is built around a stricter idea: the model proposes, the verifier decides. In the beta route, a coding LLM generates candidates, Arbiter compares and refines them with compact latent reasoning signals, and external checks decide whether a branch is trusted.

What is in the beta

The public surface is simple: an OpenAI-compatible model route named arbiter-0.1-beta. The route is designed for Orrery coding runs, repair attempts, and evidence-backed agent workflows. Arbiter adds candidate control, repair behavior, and verifier-backed selection around the LLM.

The public explanation stays intentionally high level: Arbiter uses latent reasoning. That means the system can compare the meaning and direction of possible answers before committing everything to final text. Token-by-token generation is still how code is written, but latent reasoning gives Arbiter a quieter place to weigh alternatives, avoid getting trapped by the first fluent answer, and prefer work that survives verification.

The LLM used

The premium beta route is built around Ornith 35B, a coding-focused LLM served through an OpenAI-compatible runtime. Arbiter can also run lighter Ornith-family routes for lower-cost preview availability. The point is not only the base model: Arbiter adds the verification and repair loop around it.

Short version: Ornith proposes code. Arbiter compares branches, checks evidence, repairs failures, and only treats verified work as the final answer.

Why latent reasoning helps

Tokens are excellent for expression, but they are a narrow surface. Once a model starts writing, every next token is shaped by the text already on the page. That can make a wrong path feel increasingly convincing.

Latent reasoning is useful because it works before that commitment hardens. Arbiter can compare possible directions at a more compressed meaning level, then decode and verify the stronger candidates. The advantage is not magic speed or guaranteed correctness. The advantage is better search: fewer plausible-looking dead ends, more chances to repair from evidence, and a final answer judged by what runs.

Deployment economics

Arbiter is designed to scale gradually. A lighter model route can handle early preview traffic, while the 35B route can be reserved for premium sessions, demos, and evaluation windows. GPU prices move over time, but the basic shape is stable: smaller routes are cheaper to keep warm, and larger routes should earn their extra spend with better outcomes.

Availability path

  1. Hosted route first. Arbiter appears as arbiter-0.1-beta inside Orrery rather than as a public weights download.
  2. Operational details stay server-side. The public product surface is the model route, not the underlying deployment.
  3. Access expands carefully. The beta widens as reliability, latency, and verified coding outcomes improve.
  4. Model tier follows demand. Efficient routes can serve early traffic; larger routes are promoted when they produce better verified outcomes.

Evidence boundary

Arbiter is evaluated by verified outcomes: tests passed, patches reviewed, tool calls completed, and evidence saved. Current internal beta evidence includes a 92/100 frozen local projection on the first 100 SWE-bench Verified tasks. That is not an official leaderboard score. It is an internal release signal, and benchmarking remains limited by compute capacity: on the same hardware, training, serving, and large benchmark runs compete for time, so public claims stay conservative until larger matched evaluations are complete.
